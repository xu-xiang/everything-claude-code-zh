# GitHub Copilot 使用指南

## 核心原则

在 Android Studio 中使用 GitHub Copilot 时，不要把它当成“自动完成一切的代理”，而要把它当成：

- 代码分析助手
- 草案生成器
- 测试生成器
- 重构建议助手
- 审查辅助工具

## 推荐提问模式

### 1. 先分析，再修改
不要直接说“帮我改”。优先使用：

- 先分析这个类的职责、依赖、潜在风险，再给出最小修改方案。
- 先说明根因，再给出 patch 建议。
- 先列出你准备修改的文件和原因，不要立即生成代码。

### 2. 强调最小改动
对于 Android 系统和成熟应用项目，建议固定加上：

- 优先最小改动，不做无关重构。
- 保持现有外部行为不变。
- 不引入新依赖，除非我明确要求。
- 不修改公开接口，除非我明确要求。

### 3. 指定技术上下文
根据场景补充：

- 这是 Android framework / system service 代码。
- 这是 Android app ViewModel 层代码。
- 这是 Binder / AIDL 调用链。
- 这是 Compose UI 状态管理逻辑。

### 4. 要求输出结构化
推荐要求 Copilot 用固定结构回答：

1. 问题理解
2. 根因分析
3. 最小修改方案
4. 风险点
5. 验证步骤
6. 是否需要测试补充

## 不推荐的提问方式

- “直接帮我重写这个文件”
- “优化一下”
- “改得更好”
- “帮我处理所有问题”

这些说法约束太弱，容易导致 Copilot 输出大范围无关变更。

## 建议工作流

### Bug 修复
1. 提供报错、日志、堆栈或异常现象
2. 让 Copilot 分析根因
3. 让 Copilot 提出最小修改方案
4. 人工确认
5. 再让 Copilot 生成 patch
6. 用 checklist 人工复核

### 代码审查
1. 贴出 patch 或文件
2. 要求从兼容性、安全性、性能、并发风险角度审查
3. 要求按严重程度排序输出问题

### 测试补充
1. 指定类职责
2. 指定依赖 mock 方式
3. 要求覆盖正常路径、异常路径、边界条件
