# 通用工程规则

## 目标
确保 Copilot 生成的建议更稳健、可维护、可审查。

## 规则

1. 优先最小改动，不做无关重构。
2. 保持现有外部行为不变，除非需求明确要求修改行为。
3. 不新增依赖，除非明确要求。
4. 不擅自修改公共 API、接口、协议、数据结构。
5. 修改前先分析影响范围，再给出方案。
6. 输出代码前先说明准备修改哪些文件以及原因。
7. 对不确定的业务逻辑，先提问澄清，不要猜测。
8. 优先复用项目现有模式，不引入与当前代码风格冲突的新写法。
9. 任何修复建议都应包含验证方式。
10. 如涉及风险较高的修改，先给出方案，不直接生成最终代码。

## 输出格式要求

建议 Copilot 按以下结构回答：

1. 问题理解
2. 根因或目标分析
3. 建议修改点
4. 风险说明
5. 验证方式
6. 若需要，补充测试建议
