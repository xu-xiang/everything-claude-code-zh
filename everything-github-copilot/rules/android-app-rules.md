# Android 应用开发规则

适用于：
- app 层业务代码
- Activity / Fragment / ViewModel / UseCase / Repository
- 数据流、状态管理、UI 层逻辑

## 核心规则

1. 优先保持现有用户可见行为不变。
2. 遵循现有架构边界，不跨层直接访问不该访问的模块。
3. UI 层只处理展示逻辑，不混入过多业务逻辑。
4. ViewModel 负责状态组织，不直接持有不必要的 Android UI 引用。
5. Repository 负责数据来源抽象，避免把业务决策散落到多层。
6. 对异步逻辑要明确线程、作用域、取消时机。
7. 对状态更新要避免重复触发、竞态和 UI 闪烁。
8. 不为小问题引入复杂抽象。
9. 修改 UI 时，要说明是否影响：
   - 生命周期
   - 配置变更
   - 恢复状态
   - 空态 / 错误态 / 加载态
10. 建议优先补测试或至少给出手动验证步骤。

## Compose / XML 附加要求

- Compose：关注重组成本、状态来源、remember / derivedStateOf 使用是否合理。
- XML/View：关注空指针、重复��定、生命周期泄漏、监听器解绑。
